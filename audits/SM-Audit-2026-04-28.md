## Critical Findings (must fix before launch)

### C1 — social-logic.js exceeds 60K character limit
**Category 1 — File Architecture**
File is 71,877 chars. Platform maximum is 60,000 chars per file. Must split into social-logic.js + social-modules.js.

### C2 — @anthropic-ai/sdk not in package.json
**Category 15 — Module System**
Three API files import `@anthropic-ai/sdk` (generate-social-content.js, generate-campaign-plan.js, generate-marketing-theme.js) but it is not listed in package.json dependencies. Vercel deployment may fail or use a non-deterministic version.

### C3 — meta-post.js leaks raw error messages to client
**Category 7 / API Standards — meta-post.js line 367**
`return res.status(500).json({ error: err.message })` exposes internal error details (database errors, API failures) to the browser. Must return a user-friendly message.

## Warning Findings (should fix)

### W1 — Hardcoded px values in social.html CSS
**Category 5 — Stylesheet Compliance — social.html (multiple lines)**
Numerous hardcoded pixel values: gap: 20px, padding: 24px, margin-bottom: 8px, font-size: 28px, etc. Should use CSS variables where platform equivalents exist (e.g. var(--card-padding), var(--page-header-margin), var(--heading-lg-size)).

### W2 — Inline styles in JS-built HTML strings
**Category 5 — Stylesheet Compliance**
- social-logic.js: lines 564, 618-621, 661-664, 735-736, 757-760
- social-campaign.js: lines 80, 169, 198, 225, 253, 436, 438, 575, 577, 675, 842, 844, 848, 858
- social-settings-logic.js: lines 63, 71

Layout inline styles (display:flex, margin-top, text-align:center) used in JS-built HTML. Should use named CSS classes.

- social.html line 186: `.sm-media-remove` has font-size: 12px
- social-logic.js line 564: inline font-size:12px on tone description

All below the 13px mobile minimum.

### W7 — 'we/our' language in UI copy
**Category 12 — Copy and Brand**

### W8 — Dead CSS rules
**Category 11 — Redundant Code — social.html lines 336-350**
Four CSS classes defined but never referenced: `.sm-campaign-progress`, `.sm-campaign-timeline`, `.sm-campaign-week`, `.sm-campaign-week-label`.

### W9 — meta-post.js JSDoc references wrong env var names
**Category 15 — Module System — meta-post.js lines 11-12**
Comment says `CLAUDE_API_KEY` and `SUPABASE_SERVICE_KEY` but code correctly uses `ANTHROPIC_API_KEY` and `SUPABASE_SERVICE_ROLE_KEY`. Misleading comment.

### W10 — meta-post.js uses wrong Claude model for suggest-boost
**Category 15 / API Standards — meta-post.js line 59**
Uses `claude-haiku-4-5-20251001` for the suggest-boost action. Boost suggestions are customer-facing output and should use `claude-sonnet-4-6` per spec.

### W11 — Missing maxDuration in vercel.json
**Category 15 — Deployment**
No maxDuration configured for:
- meta-post.js (AI calls via suggest-boost)
- predis-generate.js (external API)
- predis-brand.js (external API)
- social-publisher.js (sequential Meta API calls for multiple posts)
- generate-marketing-theme.js (AI call)

Default Vercel timeout is 10s which may be insufficient.

## Note Findings (minor)

### N1 — predis_generation_id column not in spec schema
**Category 13 — Database Integrity — predis-webhook.js line 26**
Column `predis_generation_id` used in `.eq()` on social_posts but not listed in the spec's social_posts schema (Section 12.2). Needs to be added when the table is created.

### N2 — bp_services column access needs verification
**Category 13 — Database Integrity — generate-campaign-plan.js line 58**
Accesses `bp.bp_services` on business_profiles. Column exists in the current schema (verified in cl-profile.js) but is not documented in the SM spec.

### N3 — 72 page-local CSS classes in social.html
**Category 14 — Component Standards**
All use the `sm-` prefix and are genuinely unique to this tool. Count is high but justified by the tool's complexity (wizard, preview, calendar, post cards, media upload).

### N4 — social-settings.html missing PWA desktop-only guard for tool page
**Category 10 — Mobile Standard**
Settings page has the PWA guard (correct — settings is desktop-only), but social.html does not have the guard (correct — social.html is mobile-capable per spec). Confirmed correct for both.

### N5 — No CL write-back implementation
**Spec Section 15 — Integration Points**
Generated blog content and marketing outputs are not written back to Content Library. Spec says blog content saves to CL tagged for website use. This is a missing feature, not a standards violation.

### N6 — Campaign history view not implemented
**Spec Section 7.7 — Campaign History**
Completed campaigns are not shown in a history view. The Campaign tab only shows the active/most recent campaign. The 'Run Again' option is missing.
